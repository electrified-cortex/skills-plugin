# publish — spec

A skill that orchestrates a new release of the skills-plugin. Bumps the plugin version, regenerates the distributed `skills/` tree via the build, updates the CHANGELOG, commits the result, tags it, and (when the host operator says so) pushes the tag and merges to `main`.

The publish skill does NOT decide WHEN to release. It executes a release the caller has already decided to cut.

## Purpose

Replace ad-hoc release ceremonies with a single skill the agent (or operator) can dispatch. Output: a tagged commit on `main` whose tree is the rebuilt skills distribution.

## Inputs

- `version_bump` — required, one of `patch | minor | major`. Determines how `plugin.json`'s `version` is incremented from its current value.
- `release_notes` — required, a human-authored summary of what changed since the last tag. Becomes the new CHANGELOG entry's body.
- `source_root` — optional override of `build/config.yaml`'s configured source path. Default: read from config.
- `dry_run` — optional boolean. When true, runs every step except the final commit/tag/push. Returns what WOULD happen.

## Outputs

- New commit on the current branch with:
  - `plugin.json` `version` bumped per `version_bump`.
  - `plugin.json` `built` set to today's UTC date (`YYYY-MM-DD`).
  - `skills/` regenerated by `build/build.ps1` against `source_root`.
  - `CHANGELOG.md` prepended with `## [<new-version>] - <today>` + `release_notes`.
- A new annotated git tag `v<new-version>` on the commit.
- Status report: previous version, new version, files changed, build report (skills count, files included, files denied).

## Procedure

1. **Pre-flight checks.** Refuse to publish if any of:
   - Working tree has uncommitted changes outside `skills/`, `plugin.json`, `CHANGELOG.md`. Stop with `dirty_working_tree`.
   - Current branch is not the configured release branch (default `main`). Stop with `wrong_branch` listing current branch.
   - `build/config.yaml` is missing or unparseable. Stop with `bad_config`.
   - Last tag's version doesn't parse as SemVer. Stop with `bad_prior_version`.

2. **Read current version.** Parse `plugin.json` for `version`. Compute new version per `version_bump`.

3. **Run the build.** Invoke `pwsh build/build.ps1`. Confirm exit code 0. Capture build report. If build emits any deny-pattern violation in `skills/<dist>/` (per `build/deny-list.ps1`'s validation pass), stop with `build_validation_failed` and surface the offending paths.

4. **Update `plugin.json`.** Set `version` to the new SemVer. Set `built` to today's date in `YYYY-MM-DD`.

5. **Update `CHANGELOG.md`.** Prepend a new section:

   ```markdown
   ## [<new-version>] - <YYYY-MM-DD>

   <release_notes>
   ```

   Preserve all prior entries verbatim.

6. **Stage + commit.** Stage exactly: `plugin.json`, `CHANGELOG.md`, `skills/` (full tree). Refuse `git add -A`. Commit message: `release: v<new-version>`. Operator may amend the message after.

7. **Tag.** `git tag -a v<new-version> -m "v<new-version>"`. The tag points at the release commit.

8. **Push (conditional).** If `dry_run` is true, stop and return the dry-run report. Otherwise push the tag and the branch:
   - `git push origin <release-branch>`
   - `git push origin v<new-version>`

   If push fails (e.g., remote diverged), stop with `push_failed` and surface git's error. Do NOT force-push.

9. **Return** the status report.

## Out of scope

- Deciding when to release.
- Bumping per-skill versions. The plugin is the unit of release; per-skill version markers, if they exist, are cache-busters set elsewhere — not by this skill.
- Building the source skills repo. This skill consumes the source tree as configured in `build/config.yaml`; it does not modify the source.
- Cross-repo coordination (notifying consumers, opening PRs in dependent repos). Future skill, not this one.
- Generating release notes. The caller supplies `release_notes`. This skill does not summarize git log.

## Bailout

- Any pre-flight check failure stops the skill before any mutation.
- `build_validation_failed` stops before any commit. The build's own validation pass (per 10-1003 acceptance criteria) is the gate.
- `push_failed` after a successful local commit/tag leaves both intact for operator inspection. Skill returns the local SHA + tag + push error verbatim.

## Dependencies

- `build/build.ps1` exists and exits 0 on success.
- `build/config.yaml` defines a valid `source` block.
- `build/deny-list.ps1` is loaded by `build.ps1` and applied during validation.
- Git working tree is on the configured release branch.
- `pwsh` (PowerShell 7+) on PATH.

## Anti-patterns

- Auto-bumping the version without an explicit `version_bump` argument. The caller chooses semver impact; the skill does not infer.
- Force-pushing the release branch on push conflict. Operator resolves divergence manually.
- Generating `release_notes` from the git log. The caller authors prose; the skill renders the CHANGELOG.
- Stamping a release whose `skills/` was hand-edited. The build is the source of truth for `skills/`.
