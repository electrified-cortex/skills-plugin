Inputs:
`skill_dir` (positional, required) — absolute path to skill folder being audited.
`--report-path <abs-path>` (required) — absolute path to write report. Missing → `ERROR: --report-path required`, stop. Existing file at `<report_path>` overwritten.

HARD PROHIBITION: DON'T author scripts (`.ps1`, `.sh`, `.py`, etc.), helper files, or any file other than `<report_path>`. Source files in `skill_dir` read-only. Use Read/Bash/Grep only for inspection.

CHECK INVENTION PROHIBITION: only evaluate checks explicitly named and defined here. Don't invent, infer, or apply unlisted checks — even if pattern looks suspicious. No named check = no finding.

Procedure:
1. Enumerate files in `skill_dir` recursively; skip dot-prefixed dirs. Tool files (`.sh`, `.ps1`) and tool-spec files (`*.spec.md` other than the skill's own `spec.md` co-located with `SKILL.md`) are out of scope for Steps 1–3 AND excluded from the manifest `file_paths` — they belong to the tool manifest (audited separately by tool-auditing). Skill manifest covers only the skill bundle: `SKILL.md`, `instructions.txt`, `spec.md`, `uncompressed.md`, `instructions.uncompressed.md` (whichever exist). Include tool/.spec.md files in A-FS-1 enumeration only.
2. Read `SKILL.md`. Determine skill type (inline or dispatch) by file-system evidence: any allowed dispatch instruction file present (`instructions.txt`, `<name>.md`, or file explicitly referenced by `SKILL.md`) → dispatch; none → inline. Locate companion `spec.md` co-located with `skill_dir`.
3. Run Per-file basic checks (always; findings → separate Per-file section; don't block Steps 1–3).
4. Run Step 1 → Step 2 → Step 3. Collect ALL findings before assigning verdict. Don't stop on first finding.
5. Assign verdict per Verdict Rules.
6. Scrub absolute paths from entire report — frontmatter `file_paths`, Notes columns, Findings, ALL prose. Forbidden tokens: Windows drive-letter paths (`<letter>:[/\\]`), POSIX root-anchored paths (`/Users/`, `/home/`, `/d/`, `/c/`, `/mnt/`, `/tmp/`, `/var/`). Evidence containing such path → describe abstractly ("hardcoded drive-letter path on line N") — never quote literal. Use repo-relative paths in Findings.
7. Write report at `<report_path>` (overwrite if present). Use Bash tool to create the directory first: `mkdir -p $(dirname <report_path>)`. Then use Write tool to write the report. Frontmatter per `hash-record` skill contract (operation_kind: `skill-auditing/v2`, result: `clean | pass | findings | fail`). Result mapping: CLEAN → `clean`; PASS → `pass`; NEEDS_REVISION → `findings`; FAIL → `fail`. Errors are never persisted — runtime/argument failures emit `ERROR: <reason>` and exit 1 (no path).

   Repo-relative: paths in `file_paths` are repo-relative as provided. If you must convert an absolute path, use `git ls-files --full-name <file>`. Same repo root governs cache dir layout (`<repo-root>/.hash-record/...`).

   Skill-auditing-specific `file_paths` guidance (multi-file context — auditors commonly miss this):

   ```yaml
   # Correct — list of repo-relative paths, one per non-tool source file from step 1, sorted lexically:
   file_paths:
     - skill-auditing/instructions.uncompressed.md
     - skill-auditing/spec.md
     - skill-auditing/uncompressed.md

   # WRONG (directory only):
   file_path: skill-auditing/

   # WRONG (absolute paths):
   file_paths:
     - /abs/path/to/skill-auditing/SKILL.md

   # WRONG (singular file_path in multi-file context):
   file_path: skill-auditing/SKILL.md
   ```

   Body: open with `# Result` H1, state verdict, list findings (with step and check refs).
8. **Verify report on disk.** Before returning a verdict, run `ls -la <report_path>` (or equivalent) and confirm the file exists with non-zero size at the EXACT path passed via `--report-path`. If missing, empty, or written elsewhere → return `ERROR: report not at <report_path>` and stop. Then scan the written report for absolute path content (forbidden tokens: Windows drive-letter paths `<letter>:[/\]`, POSIX root-anchored paths `/Users/ /home/ /d/ /c/ /mnt/ /tmp/ /var/`). If any match is found, return `ERROR: absolute path in report at <report_path>:<line>: <matched-text>` and stop. Do NOT return CLEAN/PASS/NEEDS_REVISION/FAIL without a verified, abs-path-free file.
9. Return verdict token as final stdout line, column 0, no indentation, no quoting, no list-marker prefix. Nothing follows.
   - `CLEAN: <report_path>`
   - `PASS: <report_path>`
   - `NEEDS_REVISION: <report_path>`
   - `FAIL: <report_path>`
   - `ERROR: <reason>`

Per-file Basic Checks:
Run against all `.md` and `*.spec.md` files in `skill_dir` (recursively; skip dot-prefixed dirs). Tool files (`.sh`, `.ps1`) out of scope.
Findings → Per-file section; don't block Steps 1–3.

`.md` files:
- Not empty — must contain non-whitespace content. Empty → HIGH.
- Frontmatter where required — `SKILL.md` and `agent.md` MUST have YAML frontmatter (`---` block at line 1). Missing → HIGH.
- No absolute-path leaks — body mustn't contain Windows-style (`<letter>:\` or `<letter>:/`) or Unix root-anchored paths (`/Users/`, `/home/`, `/d/`). Any found → HIGH.

`*.spec.md` files (name ends in `.spec.md`):
- Purpose section present — must contain `## Purpose` or `# Purpose`. Missing → HIGH.
- Parameters section present — must contain `## Parameters` or `# Parameters`. Missing → HIGH.
- Output section present — must contain `## Output` or `# Output`. Missing → HIGH.

Step 1 — Compiled Artifacts:
Quick structural verification of `SKILL.md` and non-tool files it references.

Classification:
"Could someone with no context do this from just the inputs?" Yes → should be dispatch. No → should be inline. Flag misclassification.

Inline/dispatch file consistency:
File presence is definitive: any allowed dispatch instruction file present (`instructions.txt` or `<name>.md` in skill dir, or file explicitly referenced by SKILL.md) → dispatch; none → inline. If Check #1 and this disagree, flag conflict but DON'T double-fail — note conflict, let auditor decide.
Dispatch → verify SKILL.md is short routing card. Inline → verify SKILL.md contains full procedure. Mismatch → FAIL.

Structure:
Inline: frontmatter (`name`, `description`), direct instructions, self-contained.
Dispatch: instruction file exists and reachable. Params typed with required/optional/defaults. Output format specified. Uses Dispatch agent (isolated), not background agent with host context. SKILL.md is minimal routing content.

Stop gates in routing card (dispatch only): if SKILL.md contains refusal conditions, eligibility guards, git-clean checks, or path-escape rules → NEEDS_REVISION. Finding text: `stop gates belong in instructions.txt, not the routing card`. Routing card = invocation signature + output format.

(A-IS-1) Input/output double-specification — if skill takes INPUT duplicating OUTPUT already determined by referenced sub-skill (e.g. passing `result_file` when sub-skill dictates output path via its own hash-record write) → HIGH. Input surface mustn't override sub-skill conventions.

(A-IS-2) Sub-skill input isolation — if a sub-skill's input surface includes a parameter that accepts the output path, result token, or computed artifact of a sibling sub-skill (e.g. a `<lint_path>` parameter consuming a sibling lint sub-skill's output), flag as HIGH. Sub-skills must be independently executable from the primary input only; the parent orchestrator owns cross-sub-skill data flow. See `skill-writing` spec R-SS-1.

Frontmatter:
`name` and `description` present and accurate.

(A-FM-1) Name matches folder — `name` MUST equal skill's folder name exactly. Check both `uncompressed.md` and `SKILL.md`; mismatch in either → FAIL.

(A-FM-4) Valid frontmatter fields — SKILL.md frontmatter MUST contain ONLY `name` and `description`. Any additional YAML key (version, inputs, outputs, type, model, tools, etc.) → FAIL.

(A-FM-11) Trigger phrases — all skills — ALL skills (inline AND dispatch): `description` MUST contain `Triggers -` (case-insensitive). Absence → HIGH. Dispatch skills additionally validated by DS-3.

(A-FM-12) `uncompressed.md` frontmatter mirror — if `uncompressed.md` exists, it MUST have YAML frontmatter. `name` and `description` MUST match SKILL.md exactly (case-sensitive). Missing frontmatter → FAIL. Mismatched `name` or `description` → FAIL.

(A-FM-3) H1 per artifact — applies ONLY to `.md` files. `.txt` files (e.g. `instructions.txt`) are NOT markdown; the H1 rule does NOT apply to them — never flag a `.txt` file under A-FM-3. For `.md` files, a "real H1" is a line that starts at column 0 and matches `^# ` literally. H1 markers INSIDE fenced code blocks (```...```), inline code (`# ...`), or quoted prose are TEMPLATE / EXAMPLE content — never count them as real H1s. To detect: scan for `^# ` AND verify the line is not inside a ``` fence (track fence toggles). Rule: `SKILL.md` MUST NOT contain a real H1. `uncompressed.md` (if present) MUST contain a real H1. `instructions.uncompressed.md` (if present) MUST contain a real H1. Violation in `SKILL.md` → HIGH. Missing H1 in `uncompressed.md` or `instructions.uncompressed.md` is out of scope — markdown-hygiene covers H1 enforcement. Absence of `uncompressed.md` is not a finding — it is optional.

No duplication:
Not duplicating existing capability. If similar exists, recommend merge or distinguish.

(A-FS-1) Orphan files:
Scan all files in `<skill_dir>` recursively; skip dot-prefixed dirs.
For each file NOT a well-known role file (`spec.md`, `*.sh`, `*.ps1`, `*.spec.md`, `eval.txt`, `*.uncompressed.md`, `SKILL.md`, `uncompressed.md`, `instructions.txt`):
- Check if referenced by name or relative path in any of: `SKILL.md`, `uncompressed.md`, `spec.md`, `instructions.uncompressed.md`. Not referenced → LOW.

Special case: `instructions.txt` present where file-system evidence shows NOT dispatch (no dispatch invocation in `SKILL.md` or `uncompressed.md`) → HIGH. Instructions file with no dispatch wiring is an orphan.

(A-FS-2) Missing referenced files:
Scan `SKILL.md`, `uncompressed.md`, `spec.md`, `instructions.uncompressed.md` for explicit file-path pointers: `instructions.txt`, `result.sh`, `result.ps1`, `verify.sh`, `verify.ps1`, any other filename literal denoting sibling file. Each must exist in `<skill_dir>`. Missing → HIGH.

Step 2 — Parity Check:
Verify each compiled artifact faithfully represents its uncompressed counterpart with no loss of intent. Minor wording compression OK; omitted requirements, contradictions, or changed behavior are not.

`uncompressed.md` is optional. Its absence is not a finding. When absent, the SKILL.md ↔ uncompressed.md parity row is N/A.

| Compiled | Uncompressed counterpart |
| --- | --- |
| `SKILL.md` | `uncompressed.md` (optional) |
| `instructions.txt` | `instructions.uncompressed.md` |

Both exist → read both, compare intent. Compiled diverges → finding text: `Parity failure: <description>. Fix: edit <uncompressed-file>, then recompress to <compiled-file>.`
No counterpart → skip (N/A).

Advisory (LOW, non-blocking): if `uncompressed.md` is absent and `SKILL.md` is >60 lines → suggest maintaining an uncompressed source for readability and safe editing. Do not count toward verdict severity.

Step 3 — Spec Alignment:
Verify companion spec is structurally sound, then verify compiled artifacts faithfully represent it.

Spec exists:
`spec.md` must exist co-located with `skill_dir`. Exception: simple inline skills (<30 lines) may skip entirely.
Not found and skill is dispatch or complex inline → FAIL; continue collecting findings from Steps 1–2.

Required sections:
Spec must contain: Purpose, Scope, Definitions, Requirements, Constraints. Missing → FAIL.

Normative language:
Requirements must use enforceable language (must, shall, required). Vague terms → FAIL.

Internal consistency:
No contradictions between sections. No duplicate rules. No normative content in descriptive sections.

Spec completeness:
All terms defined. All behavior explicitly stated, not implied.

Coverage:
Every normative requirement in spec must be represented in SKILL.md. Missing → FAIL.

No contradictions:
SKILL.md mustn't contradict spec. Spec authoritative; SKILL.md subordinate.

No unauthorized additions:
SKILL.md mustn't introduce normative requirements absent from spec.

Conciseness:
Every line affects runtime behavior. No rationale in SKILL.md (belongs in spec). No redundant explanations. Agent-facing density.

Named patterns to flag:
- "Too much why": prose explaining why rule exists. Finding text: `too much why — move rationale to spec.md`.
- "Essay not reference card": reads as continuous prose rather than decision trees, tables, or bullets. Finding text: `exposition where a decision tree would serve`.
- "Prose conditionals": if/else logic as prose paragraphs. Finding text: `replace prose conditionals with decision tree or table`.
- "Meta-architectural label": line describing skill's own execution pattern rather than instructing agent. Finding text: `meta-architectural label — remove, this is design rationale`.

Passes conciseness only if agent can skim in one pass and know exactly what to do.

Skill completeness:
All runtime instructions present. No implicit assumptions. Edge cases addressed or excluded. Defaults stated.

Breadcrumbs:
Ends with related skills/topics. References valid (targets exist). No stale references.

Cost analysis (dispatch only):
Uses Dispatch agent (zero-context isolation). Instruction file right-sized (<500 lines). Sub-skills referenced by pointer, not inlined. Single dispatch turn when possible.

No dispatch references in instructions:
`instructions.txt` mustn't tell agent to dispatch other skills. Subagents can't dispatch — only host agent can. References as "Related" context OK. "Run this skill" or "dispatch this" → FAIL. Remediation: move dispatch steps to SKILL.md.

No spec breadcrumbs in runtime:
SKILL.md and `instructions.txt` mustn't reference skill's own companion `spec.md` — not as pointer, breadcrumb, or "see spec.md" hint. Exception: skills whose operation takes spec as input (`spec-auditing`, `skill-auditing`) may reference `spec.md` under audit — never their own. Remediation: delete reference; if info genuinely needed at runtime, inline it.

Eval log presence (informational):
Perform eval-presence check by reading co-located `eval.txt` sub-instructions. Full procedure in `eval.txt` / `eval.uncompressed.md`. Absence of `eval.md` MUST NOT affect verdict.

(A-FM-2) Description not restated:
Search every artifact (`uncompressed.md`, `SKILL.md`, `instructions.uncompressed.md`, `instructions.txt`) for body prose duplicating `description` frontmatter value. Any restatement → LOW (escalate to HIGH if verbatim).

(A-FM-5) No exposition in runtime artifacts:
Scan `SKILL.md`, `uncompressed.md`, `instructions.uncompressed.md`, `instructions.txt` for rationale, "why this exists," root-cause narrative, historical notes, or background prose. Any found → HIGH. Rationale belongs exclusively in `spec.md`.

(A-FM-6) No non-helpful tags:
Check all artifacts for descriptor lines with no operational value (e.g., "inline apply directly no dispatch," "dispatch skill," bare type labels not used as actionable instructions). Any found → LOW.

(A-FM-7) No empty leaves:
Empty section = leaf heading with no body AND no subheadings before the next heading at the same level or higher (or EOF) → HIGH. Headings with subsections are never empty.

(A-FM-8) Iteration-safety placement:
Verify Iteration Safety blurb absent from `instructions.uncompressed.md` and `instructions.txt`. Presence in either → HIGH. Guard in both `SKILL.md` and `instructions.*` → probable duplication → HIGH.

(A-FM-9a) Iteration-safety pointer form:
If caller skill references iteration-safety, verify exact 2-line pointer block form (`Do not re-audit unchanged files.` + `See <path>/iteration-safety/SKILL.md`.`) and relative path matches caller's actual folder depth. Any deviation → HIGH.

(A-FM-9b) No verbatim Rule A/B restatement:
Scan all artifacts for verbatim restatement of iteration-safety Rules A or B beyond sanctioned 2-line pointer block. Any found → HIGH.

(A-XR-1) Cross-reference anti-pattern:
Per R-FM-11, every cross-skill reference MUST identify the target by canonical `name`. A relative path MAY follow the name as an optional pointer. References that supply ONLY a path with no canonical name are violations.

Scan `SKILL.md`, `instructions.txt`, any sub-instructions (e.g. `eval.txt`), `instructions.uncompressed.md`, `uncompressed.md`, and `spec.md`.

Violations (path with no canonical name → HIGH):
- "See ../compression/uncompressed.md for details" (path, no name)
- "Consult ../spec-writing/spec.md for the format" (path, no name)
- Any inline link to `.uncompressed.md` or `.spec.md` of another skill with no name reference

NOT violations:
- Bare name: "the `compression` skill"
- Name + folder pointer: "the `compression` skill (`../compression`)"
- Name + specific-file pointer: "the `compression` skill's tier rules (`../compression/<tier>/rules.txt`)"
- References to OWN sub-files within the same skill folder
- Subject-matter mentions of these file types in skill-auditing's own artifacts (skill-auditing audits these files as targets, so it must name them)

Any other cross-file pointer with no canonical name → HIGH.

(A-IR-1) Input redefinition in instructions:
Scan all instructions files in `skill_dir`: `instructions.txt`, `instructions.uncompressed.md`, and shell-split variants (`instructions.<shell>.txt`, `instructions.<shell>.uncompressed.md`).
For each: check whether it contains a heading introducing an inputs block — `## Inputs`, `# Inputs`, or an equivalent parameter table (block of lines with param names + type annotations or dashes under an inputs/parameters heading). If found, verify `SKILL.md` or `uncompressed.md` already declares those same params. If already declared → flag HIGH.
Applies equally to router skills (whose `SKILL.md` says "read and follow `instructions.<ext>`"): instructions file is a continuation of the `SKILL.md` context; restating inputs is decorative bloat.
Finding text: `A-IR-1: instructions file '<filename>' restates input parameters already declared in SKILL.md. Remove the Inputs section from the instructions file.`

(A-IR-2) Return contract redefinition in instructions:
Scan all instructions files: `instructions.txt`, `instructions.uncompressed.md`, and shell-split variants.
For each: check for return/output section — headings `## Return`, `# Return`, `## Returns`, `# Returns`, `## Output`, or `# Output`. If found, compare against return declaration in `SKILL.md` or `uncompressed.md`. Verbatim or near-verbatim match → flag HIGH. Material extensions adding genuinely new error cases or output modes not in `SKILL.md` are permitted — don't flag those.
Finding text: `A-IR-2: instructions file '<filename>' restates the return contract already declared in SKILL.md. Remove the redundant Return/Output section from the instructions file.`

(A-IR-3) Frontmatter leak in instructions:
Scan all instructions files: `instructions.txt`, `instructions.uncompressed.md`, and shell-split variants.
For each: check whether the first non-empty line is `---` (YAML frontmatter opening delimiter). If yes → flag HIGH.
Frontmatter belongs only in `SKILL.md`, `uncompressed.md`, agent files, and tool-spec files. Instructions files are runtime executor content — frontmatter in them is never consumed.
Finding text: `A-IR-3: instructions file '<filename>' has YAML frontmatter (--- at line 1). Remove the frontmatter block; it belongs in SKILL.md or uncompressed.md only.`
If skill has no instructions files → A-IR-3 is N/A.

(A-FM-10) Launch-script form on dispatch skills:
N/A for inline skills. N/A if `uncompressed.md` is absent (SKILL.md-only is valid).
If `uncompressed.md` is present for a dispatch skill, verify it contains ONLY:
- Frontmatter (`name`, `description`)
- Optional H1
- Dispatch invocation + input signature (parameter list)
- Return contract (`Returns:` line)
- Optional 2-line iteration-safety pointer
- Optional inline result check protocol (pre-dispatch cache check + post-execute result routing via co-located result tool)

Any of following → HIGH:
- Executor procedure steps or phase descriptions
- Modes tables or "When to audit which artifact" tables
- Examples or sample invocations beyond input signature
- Rationale, "why this exists," or background prose
- Related breadcrumbs or "Related:" sections
- Behavior sections or Output format descriptions
- Model-class guidance (e.g., "Haiku-class handles most audits")
- False-positive guard lists

Content must go in `instructions.uncompressed.md` (executor procedure) or `spec.md` (rationale/behavior).

Dispatch Skill Checks (DS-1..DS-8):
N/A for inline skills. Run against `uncompressed.md` (host-facing card).

(DS-1) Return shape declared:
Host card MUST explicitly declare return shape. Canonical: `PATH: <abs-path-to-artifact>` on success, `ERROR: <reason>` on pre-write failure. Card returning content (multi-line reports, structured findings, inline results) instead of path — for skill producing artifact — MUST be flagged. Missing → HIGH.

(DS-2) Host card minimalism:
The host card (`uncompressed.md` when present; `SKILL.md` for SKILL.md-only dispatch skills) MUST NOT contain any of following; each present → HIGH:
- Internal cache mechanism descriptions (e.g., "iteration-safe via hash-record", "hash blob check") — caching is impl detail invisible to host. EXCEPTION: A-FM-10 inline result check protocol sections are explicitly allowed host behavior — do NOT flag as DS-2 violation.
- Adaptive/conditional rules invisible to host (e.g., "MD041 auto-suppressed when frontmatter present") — belong in `instructions.uncompressed.md`.
- Tool-fallback hints (e.g., "use the CLI if available") — belong in instructions where dispatched agent decides.
- Subjective qualifiers (e.g., "Sonnet-class equivalent or diminishing returns") — replace with operative model class in dispatch line.
- Prose describing what skill does or how it works internally — belongs in `spec.md`. Card answers: how to dispatch, what params, what return shape, what modes. Nothing else.

(DS-3) Description trigger phrases:
Frontmatter `description` MUST follow pattern: `<one-line action>. Triggers - <phrase1>, <phrase2>, ..., <phraseN>.` with comma-separated trigger phrases. Trigger phrases MUST be present. Violations:
- No trigger phrases at all (prose without any `Triggers -` block) -> HIGH. Skill is undiscoverable.
- Trigger phrases stuffed with impl notes (e.g., "Dispatch skill", "Iteration-safe via hash-record", "Zero errors gate") -> HIGH. Waste trigger budget, must be removed.
- Phrase count is not policed.

(DS-4) Inline dispatch guard:
Dispatch skills MUST use canonical prompt-only dispatch pattern. See `dispatch/dispatch-pattern.md` for context.
Required elements in `uncompressed.md`:
- `<instructions>` binding MUST include `NEVER READ` guard (e.g. `NEVER READ THIS FILE` or `NEVER READ`) on same line.
- `<prompt>` binding MUST use `Read and follow <instructions-abspath>` form.
- MUST delegate via `Follow dispatch skill. See <path>/dispatch/SKILL.md`.

Violations:
- Missing `NEVER READ` guard on `<instructions>` binding → HIGH.
- Missing `<prompt>` binding or not using `Read and follow` form → HIGH.
- Missing `Follow dispatch skill. See ...` delegation line → HIGH.
- Stale standalone opener (`Without reading... yourself, spawn...`) or closer (`NEVER READ OR INTERPRET...`) outside variable block → MEDIUM (stale pattern; remove).

(DS-5) No substrate duplication:
Consumer skills producing records (audit reports, hygiene reports, review reports) MUST NOT inline hash-record path schema, frontmatter shape, or shard layout from referenced substrate skill (e.g., `hash-record`). Reference substrate by name only. Duplication of path math, frontmatter schema, or shard layout → HIGH.

(DS-6) No overbuilt sub-skill dispatches for trivial work:
Sub-skill whose entire procedure fits within 2–3 inline steps in consumer's instructions is anti-pattern. Flag any sub-skill a consuming agent could execute directly in 2–3 steps → LOW (escalate to HIGH if sub-skill adds no logic beyond filesystem op plus write).

(DS-7) Tool integration alignment:
For skills shipping a co-located tool trio (`<stem>.sh` + `<stem>.ps1` + `<stem>.spec.md`), check integration WITHOUT auditing the tool (tool-auditing does that):
- Orphan tool — every tool present in `skill_dir` MUST be referenced by `SKILL.md` or `spec.md` (by stem name or relative path). Unreferenced tool present → HIGH.
- Missing tool — every tool referenced by `SKILL.md` or `spec.md` MUST exist in `skill_dir` as a complete trio. Referenced tool absent or incomplete trio → FAIL.
- Tool-spec alignment — IF a referenced tool has `*.spec.md`, its declared behavior (Purpose, Output, return contract) MUST be consistent with how `SKILL.md` / `spec.md` describes the tool's role. Contradiction (main spec says "moves A to B"; tool spec says "deletes") → FAIL.
Auditor reads tool-spec text for this check; tool files remain excluded from manifest hash per Step 1.

(DS-8) Canonical trigger phrase:
N/A for inline skills. N/A for sub-skill files (applies only to the top-level `SKILL.md`).
For dispatch skills (frontmatter `type: dispatch`): `description` MUST include the canonical phrase = folder name with hyphens replaced by spaces (e.g., `spec-auditing` → "spec audit", `tool-auditing` → "tool audit", `markdown-hygiene` → "markdown hygiene"). Check case-insensitively. A trigger of the form `<verb> <root>` or `<root>` alone MUST appear verbatim in the description triggers list.
Missing phrase → HIGH (DS-8). Finding text: `Canonical trigger phrase '<phrase>' missing from description triggers.`

These checks extend Step 3. Violations recorded in Step 3 findings table under "Dispatch Skill Checks" group. HIGH violations → NEEDS_REVISION or FAIL depending on count; LOW → NEEDS_REVISION.

Verdict Rules:
- CLEAN: All steps pass with zero findings (no HIGH, no LOW, no informational).
- PASS: No FAIL findings, no HIGH findings. Non-blocking findings (LOW, informational) may be present.
- NEEDS_REVISION: No FAIL findings, but HIGH or multiple LOW present. List specific fixes.
- FAIL: Any FAIL finding, or 3+ HIGH findings.
