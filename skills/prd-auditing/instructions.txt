PRD Auditing — Executor Instructions

Audit PRD in two phases: spec-auditing first, then PRD-specific checks. Write combined report to `<report_path>`.

Phase 1 — Base Spec Audit:
Read and apply `../spec-auditing/instructions.txt` against `prd_path`. Treat PRD as spec under audit. Collect Phase 1 findings into buffer; don't write report yet.
If Phase 1 returns executor ERROR, surface error and stop without writing.

Phase 2 — PRD-Specific Checks:
After Phase 1 completes, apply checks below. Accumulate findings.

Step A — Required PRD Sections:

| Section | Severity if missing |
| --- | --- |
| Header (title, author, status, version, date) | HIGH per missing field |
| Summary (≤ 5 sentences, problem + product + outcome) | FAIL if absent; HIGH if length or content fails |
| Problem (audience + cost of status quo) | FAIL if absent |
| Users / Personas (or explicit exemption sentence) | HIGH if absent without exemption |
| Goals and Success Metrics | FAIL if absent |
| Functional Requirements | FAIL if absent |
| Non-Functional Requirements | HIGH if absent |
| Out of Scope | FAIL if absent |
| Assumptions, Constraints, Dependencies | HIGH if absent |
| Open Questions (section may be empty) | LOW if absent |
| Release / Rollout Notes | LOW if absent |

Step B — Content Discipline:
1. Outcome vs output — each Goal expresses measurable change. Output-as-goal → HIGH.
2. Goal-Metric pairing — each Goal pairs with quantified Success Metric + target, threshold, or direction. Missing pairing → HIGH.
3. AC coverage — each Functional Req has ≥1 Acceptance Criterion. Missing → HIGH.
4. Binary AC — each AC is pass/fail and verifiable from document text. Non-binary → HIGH.
5. Subjective qualifiers — scan ACs and NFRs for "fast", "easy", "intuitive", "responsive", "robust", "should feel" or equivalents. Any occurrence → HIGH.
6. NFR measurability — each NFR states measurable threshold or compliance standard. Missing → HIGH.
7. Out-of-Scope discipline — ≥1 explicit exclusion when product has adjacent capability plausibly confused with it. Empty when warranted → HIGH.
8. Open Question hygiene — each blocking item names decider (role or person) + blocking effect; non-blocking items marked. Blocking without decider → HIGH.
9. Internal consistency — no Functional Req contradicts Out-of-Scope; no Goal contradicts Constraint. Any contradiction → FAIL.

Step C — Scope Boundary:
1. No implementation detail — architecture, API design, schemas, library choices in FRs/NFRs → HIGH.
2. No market analysis — TAM, SAM, competitive landscape → HIGH.
3. No business case — ROI, staffing justifications → HIGH.
4. No pixel-perfect UX — embedded wireframes, pixel-precise mockups → HIGH. Conceptual flow descriptions permitted.
5. No sprint backlog — story points, assignments, ticket breakdowns → HIGH.
6. No staffing detail — RACI, team rosters → HIGH.

Step D — Banned Terminology:
1. "non-goals" — any occurrence → HIGH. Recommend rename to "Out of Scope".

Verdict:
Combine Phase 1 and Phase 2 findings:
- CLEAN — Phase 1 CLEAN AND no Phase 2 findings.
- PASS — Phase 1 PASS or CLEAN AND no HIGH or FAIL in Phase 2.
- NEEDS_REVISION — no FAIL in either phase, but any HIGH in Phase 2 OR 2+ LOW findings combined.
- FAIL — Phase 1 FAIL, OR any Phase 2 FAIL, OR 3+ combined HIGH findings.

When in doubt between PASS and NEEDS_REVISION, choose NEEDS_REVISION.
