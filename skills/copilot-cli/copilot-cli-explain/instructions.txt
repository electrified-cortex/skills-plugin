## Prerequisites

```bash
copilot --version   # must resolve; fail-fast if not
```

Fails → return `Status: UNAVAILABLE`; surface stderr; stop. Do NOT attempt installation.

## Invocation

```bash
copilot -p "<prompt>" -s --allow-all-tools
```

Add `--model <model>` only when caller explicitly supplied model name. Omit otherwise; do not pin model.

> **Note:** Named models (e.g. `gpt-4o`, `gpt-4.5`) may be unavailable via `--model`. Use default unless caller requires named model and it's confirmed available.

Constrain working directory to repo containing target file — never `/`, `~`, or directories containing secrets.

## Prompt Construction

Frame prompt to request explanation. Serialize all content inline — Copilot CLI has no file-input flag:

```text
Explain the following code. Describe what it does, why it works that way, and any
non-obvious behavior. Be concise.

<inline code content>
```

Replace `<inline code content>` with caller-supplied content as string.

## Output Shape

```text
Status: OK | ERROR | UNAVAILABLE
Explanation: <Copilot's markdown explanation>
```

| Status | Condition |
| --- | --- |
| `OK` | Copilot returned a response |
| `ERROR` | Binary returned non-zero exit code |
| `UNAVAILABLE` | `copilot --version` failed before invocation |

## Error Handling

| Condition | Action |
| --- | --- |
| `copilot --version` fails | `Status: UNAVAILABLE`; surface stderr; stop |
| Model unavailable | `Status: ERROR`; surface "model not available"; stop |
| Copilot exits non-zero | `Status: ERROR`; surface stderr; stop |

## Rules

- Return explanation verbatim in `Explanation` field — do not summarize or reinterpret.
- One code region per invocation.
- Constrain working directory to target file's repo.
